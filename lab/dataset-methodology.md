# Методика сбора лабораторного датасета

Цель методики - получить воспроизводимый датасет сетевых событий, где каждое событие можно связать с известным сценарием, временным окном, источником, целью и ожидаемой меткой.

## Принципы

1. Все прогоны выполняются только в изолированной лабораторной сети.
2. Внешние адреса запрещены.
3. Каждый сценарий описывается YAML-манифестом.
4. Для сценариев задаются длительность, интенсивность, allowlist целей и ограничения безопасности.
5. Сырые логи и PCAP не хранятся в Git.
6. Разметка строится по временным окнам сценариев и сохраняется отдельно от сырых событий.

## Зачем нужен обычный фоновый трафик

Модель обнаружения инцидентов не должна учиться только на искусственно выделенных атаках. В реальной сети подозрительная активность смешана с обычными действиями пользователей и сервисов. Поэтому датасет должен включать HTTP-запросы, DNS-активность, обращения к API, административные подключения и скачивание небольших файлов.

Фоновый трафик нужен для оценки ложных срабатываний. Без него модель может завышать качество на тестах, но плохо работать на обычной активности.

## Почему сценарии атак вплетаются в обычную активность

Контролируемые attack-сценарии запускаются на фоне benign-активности, чтобы приблизить датасет к реальному потоку событий. Это помогает проверять:

- устойчивость признаков;
- качество временной разметки;
- смешение классов;
- влияние шума на confidence модели;
- долю ложных срабатываний и пропусков атак.

## Формирование датасета

1. Подготовить план прогона.
2. Запустить обычный фоновый трафик.
3. Выполнить контролируемые сценарии по расписанию.
4. Собрать логи Zeek и Suricata.
5. Нормализовать события в JSONL.
6. Сформировать manifest временных окон.
7. Объединить события с метками.
8. Построить агрегированные признаки.
9. Создать отчет о датасете.

## Ограничения v0.1

Docker-стенд v0.1 является прототипом. Он не моделирует полностью поведение корпоративной сети и не заменяет отдельный стенд на виртуальных машинах. Полученный датасет должен рассматриваться как лабораторный материал для отладки pipeline, а не как финальный датасет для промышленной оценки.

## Выполнение и нормализация

После создания natural manifest сценарии можно выполнить по manifest. В v0.1 поддерживаются два режима:

- обычное выполнение внутри Docker-сети с обращениями только к `target-web`, `target-api` и `control-api`;
- mock-режим, который не выполняет сетевые обращения, но записывает служебные события выполнения и синтетические учебные traffic events.

Результат выполнения сохраняется в двух уровнях:

- `execution_events.jsonl` - служебный журнал выполнения сценариев;
- `traffic_events.jsonl` - учебные события сетевой активности внутри сценариев.

Затем оба файла нормализуются в `filin/lab/output/normalized_events.jsonl`, после чего формируется `dataset_report.md`.

Mock-режим формирует синтетические лабораторные события и нужен для проверки pipeline. Для обучения итоговых моделей требуется реальный сбор трафика в Docker/VMware-стенде и последующая проверка качества датасета.

## Построение признаков

Сырые события не являются готовыми признаками для модели. Для обучения используются агрегированные window-level и flow-level датасеты:

```text
raw events -> normalized events -> feature extraction -> windows.csv / flows.csv -> training
```

Window-level датасет строится по временным окнам наблюдения. Flow-level датасет v0.1 группирует события по роли источника, роли цели, сценарию и типу события. После подключения Zeek/Suricata flow-level представление должно быть расширено реальными сетевыми flow-полями.

`scenario_id`, `run_sequence` и planned time используются только для разметки и анализа, но не являются входными признаками модели. Эти поля сохраняются как metadata и должны исключаться перед обучением.
