# Протокол v0.3.17: длительная контролируемая локальная репетиция

Revision 1 была зафиксирована до первого события, но после старта признана недействительной: синхронный fsync каждой raw feature/prediction строки создал устойчивое processing lag. Кампания остановлена на 1 512-м capture-сегменте, сохранена без переписывания и не используется как evidence.

Revision 2 устранила вычислительное отставание, но также признана недействительной после старта. Sensor-runtime открывал новое TLS-соединение для каждого небольшого batch, поэтому при нулевом processing lag возник устойчивый transport lag: на 1 813 закрытых capture-окнах source содержал 13 450 canonical events, а durable connector и receiver — по 12 710. Runtime сохранён без переписывания и не используется как evidence.

Revision 3 — `ml/protocols/v0_3_17_protocol_r3.yaml` — наследовала неизменяемые schedules, durations, gates и readiness базового protocol по SHA-256, меняла только жизненный цикл sensor→connector TLS-соединения и использовала новые run/session IDs, seeds, runtime IDs и synthetic certificates.

Revision 3 завершила полные 5 400 секунд endurance run, но не опустошила backlog за разрешённые 600 секунд: source создал 41 340 canonical events, durable connector и receiver сохранили по 24 420. Причиной оказалось отдельное новое TLS-соединение connector→receiver для каждого live batch; с ростом журнала это увеличивало конкуренцию trace/checkpoint операций. Runtime сохранён и не используется как evidence.

Revision 4 — `ml/protocols/v0_3_17_protocol_r4.yaml` — дополнительно переводила connector→receiver на постоянное последовательное mTLS/TLS 1.3 соединение каждого из двух неизменных delivery workers. Все научные и эксплуатационные критерии, schedules, durations и readiness оставались неизменными; повтор получил новые run/session IDs, seeds, runtime IDs и synthetic certificates.

Официальная campaign revision 4 не запускалась. Pre-campaign live smoke подтвердил 12 000/12 000 с drain 30 секунд, но продолжение на той же базе выявило рост хвоста до 1 000 событий. Причиной были полный scan/sort неиндексированного pending-запроса каждые 100 ms и размещение санитарного trace в одном WAL с contractual durable rows. Smoke сохранён и не используется как evidence.

Revision 5 — `ml/protocols/v0_3_17_protocol_r5.yaml` — сохраняла постоянные TLS-соединения обеих границ, добавляла индекс `(delivery_status, journal_durable_ns, event_id)` для pending pump и выносила observability trace в отдельные локальные SQLite WAL-файлы. Journal, checkpoint, receiver commit, ACK и все frozen contracts оставались неизменными. Кампания использовала новые IDs, seeds, runtime namespace и сертификаты.

Revision 5 успешно завершила run A: 5 400,000 фактических секунд, 5 400 capture-сегментов и 41 340/41 340 canonical events без потерь. На границе runs до первого capture run B выявлено повторное подключение общего каталога volumes: новый receiver уже содержал события run A. Это нарушило independence gate, поэтому вся revision 5 сохранена и исключена из evidence.

Авторитетная текущая revision 6 — `ml/protocols/v0_3_17_protocol_r6.yaml`. Она сохраняет все transport/storage исправления revision 5 и меняет только bind-маршрутизацию: sensor, connector и receiver каждого run используют собственный каталог `<run_id>/volumes`. Предыдущий run никогда не монтируется в следующий; все IDs, seeds, runtime namespaces и сертификаты снова новые.

## Назначение и границы

Этап проверяет только длительную техническую устойчивость неизменного кандидата `v03154:65a3dd912d845bc1` и изолированного staging transport stack. Проверка выполняется на новых синтетических данных в локальных internal Docker-сетях при переменной нагрузке, обслуживании, перезапусках, backpressure и контролируемом давлении на выделенные runtime-хранилища.

Это не реальный shadow mode, не backend integration, не production, не внешняя validation и не научная переоценка модели. Fit, calibration/conformal fit, feature/threshold selection, labels, реальные данные, внешняя сеть, опубликованные порты, automatic actions и network blocking запрещены.

## Историческая и кандидатная неизменность

До freeze вычислены агрегированные SHA-256 официальных артефактов v0.3.11–v0.3.16, включая обе revision v0.3.16 и v0.3.14 errata. Полный набор значений и метод агрегации зафиксированы в протоколе. Особо закреплены backend tree `04218a4eb01534950efd5f7d6390f1a575cacbc8`, candidate artifact `65a3dd912d845bc1d6e44247bb8b98fe228a7a4e0496d56a73857febbaa4df87`, manifest `56d95a75b6ce5a81a3bd5366245b3adf98314de59bdd44f13bdd138f2ddf3537`, `network_features_v2`, оба shadow-event contracts, registry и commitment. Они повторно проверяются после завершения этапа.

Предложенные seeds `24101`, `24201`, `24301` уже использовались недействительной revision 1 этапа v0.3.16. До freeze они заменены свободными run seeds `27101`, `27201`, `27301`; двенадцать отдельных session seeds также заморожены в протоколе. Совпадения проверялись по объявлениям seeds официальных этапов, а случайные цифровые фрагменты SHA-256 не считались seed.

## Архитектура и изоляция

Обязательны пять раздельных компонентов: `traffic-source`, `sensor-runtime`, `staging-connector`, `reference-receiver`, `operator-view`. Они соединяются тремя internal-сетями: sensor–connector, connector–receiver и receiver–operator. Внешние маршруты, host networking, published ports и backend route отсутствуют. Контейнеры работают non-root с read-only root filesystem, `no-new-privileges`, `cap_drop: ALL`, scoped volumes, лимитами и healthchecks.

Operator view является отдельным read-only представлением над неизменяемыми receiver snapshots. Оно поддерживает только `GET` и `HEAD`, не имеет write credentials, writable database и action controls. Ровно разрешённые поля `operator_projection_v1` перечислены в frozen protocol; raw identifiers, payload, features, labels, secrets, сценарная семантика и рекомендации действий запрещены. Observability следует отдельному санитарному контракту `rehearsal_observability_v1`.

## Фактическая длительность и нагрузка

Кампания состоит из трёх независимых runs общей фактической длительностью не менее 14 400 секунд:

- Run A `local_rehearsal_endurance_r2_001`: 5 400 секунд непрерывного nominal endurance;
- Run B `local_rehearsal_transition_r2_001`: 5 400 секунд переходов нагрузки и обслуживания;
- Run C `local_rehearsal_recovery_r2_001`: 3 600 секунд slowdown, backlog, disk pressure и recovery.

Ускоренные часы, изменение системного времени, mocking ожидания и timestamp-only simulation запрещены. Каждую фактическую секунду закрывается как минимум одно неизменяемое causal capture window; минимум — 14 400 окон. Первые 60 окон каждого run являются warmup и проходят тот же pipeline, но не входят в canonical event set. Точные фазы, rates, maintenance, rotations, restarts, faults, storage thresholds и recovery interval заданы в протоколе.

Нагрузка включает восемь семейств синтетического поведения и четыре уровня: idle `0–1`, nominal `5–15`, elevated `20–50`, bounded burst `100` событий/с. Burst длится не более 120 секунд, направлен только во внутренний synthetic receiver и не является DoS.

## Обслуживание, отказоустойчивость и доказательства

До первого события заморожены две фактические ротации синтетических mTLS-сертификатов, семь restart-сценариев, isolated-volume disk pressure, три режима receiver slowdown, bounded queues/journals/outbox, 30 fault/maintenance scenarios и 24 security negative tests. ACK до durable commit запрещён; после recovery backlog обязан стать нулевым. Старые сертификаты отклоняются после 60-секундного grace period.

Resource samples собираются каждые 10 секунд; для каждого постоянно работающего компонента требуется не менее 1 440 samples. Raw PCAP, Zeek logs, feature rows, predictions, events, journals, databases, snapshots, ключи, resource и latency traces остаются только в gitignored `runtime/v0_3_17`. В Git допускаются только санитарные aggregates, manifests, hashes и отчёты.

Множества sensor source, connector durable/acknowledged и receiver durable unique должны совпасть. Operator projection обязана быть полной для frozen projection policy. Проверяются шесть hash/commit chains, clock domains, latency breakdowns, availability, resource trends, compaction, privacy, strict resume и двадцать corruption cases. Policy result не может доказывать сам себя: каждое утверждение связывается с независимым evidence через claim-evidence ledger.

## Freeze, code lock и invalidation

После этого коммита запрещено менять длительности, transitions, IDs, seeds, schedules, thresholds, operator fields, gates и readiness. До первого rehearsal event должны быть завершены реализация, validators и tests, после чего создаётся `pre_campaign_code_lock.json` с HEAD, hashes кода, protocol, contracts, images, compose и schedules.

Ошибка до первого события требует сохранённой новой protocol revision. Ошибка после первого события или любое изменение кода после старта инвалидирует кампанию: старые данные сохраняются, а повторный запуск получает новые session IDs, seeds, runtime IDs и synthetic certificates.

Положительный результат разрешает только `candidate_ready_for_v0_3_18_external_review_and_trial_design=true`, то есть подготовку отдельного design-review. Shadow mode, backend integration, production, внешние подключения, реальные данные, automatic enforcement и notifications остаются запрещены независимо от результата.
# Актуализация protocol revision 7

Revision 6 завершила первый 90-минутный run без потерь, но была инвалидирована
при переходе между runs. Независимые durable volumes были пустыми, однако новый
контейнер `traffic-source` прочитал оставшийся `control.json` первого run раньше,
чем оркестратор опубликовал управление вторым run, и начал повторно дописывать
capture первого run. Кампания остановлена до первого capture второго run,
сохранена в `runtime/v0_3_17_invalidated_r6` и не используется как evidence.

Авторитетная текущая revision 7 — `ml/protocols/v0_3_17_protocol_r7.yaml`.
Перед запуском очередного `traffic-source` оркестратор удаляет устаревшие
`control.json` и `source_last_completion.json`; новый control публикуется только
после готовности нового изолированного стека. Все run/session IDs, seeds,
runtime namespaces и синтетические сертификаты заменены. Candidate, contracts,
schedules, фактические длительности, gates и readiness не изменены.
# Актуализация protocol revision 8

Revision 7 успешно завершила независимый run A, однако run B выявил два
эксплуатационных ограничения. Вторая certificate rotation завершилась
`passed=false`: receiver был принудительно остановлен с exit code 137 и не
возобновился. Кроме того, SQLite `FULL` durability на Windows bind mount не
успевала опустошать очередь переходной нагрузки. Runtime сохранён в
`runtime/v0_3_17_invalidated_r7` и полностью исключён из evidence.

Авторитетная текущая revision 8 — `ml/protocols/v0_3_17_protocol_r8.yaml`.
Каждый run получает три отдельных локальных Docker named volumes, а проверяемые
offline snapshots копируются в evidence root run до остановки контейнеров.
Ротация заменяет synthetic certificate files атомарно и выполняет явный
`stop`/`up` с проверкой здоровья. Durable schemas, `FULL` commit, contracts,
candidate, schedules, длительности, gates и readiness остаются неизменными.

# Итог revision 8

Кампания revision 8 завершила три независимых запуска общей фактической длительностью
`14400.000291072` секунды. Захвачено `201600` закрытых синтетических окон,
